# LexIQ API Reference

This document outlines the REST API and Server-Sent Events (SSE) endpoints provided by the LexIQ FastAPI backend.

## Base URL
Local Development: `http://localhost:8000`

## Endpoints

---

### 1. System Health
#### `GET /health`
Liveness and readiness probe to check if the API is running and the AI engine is fully initialized.

**Response (200 OK)**
```json
{
  "status": "ok",
  "engine": "LexIQ v3",
  "initialized": true,
  "sessions": 42
}
```

---

### 2. Root Entry
#### `GET /`
Basic connection test.

**Response (200 OK)**
```json
{
  "message": "LexIQ API is running. POST /chat to query."
}
```

---

### 3. Streaming Chat Endpoint
#### `POST /chat`
The core endpoint of LexIQ. Evaluates user queries through an agentic LLM pipeline and returns a real-time SSE stream of tool statuses, text tokens, and metadata.

**Request Body Schema (`ChatRequest`)**
```json
{
  "query": "string (1-2000 chars, required)",
  "session_id": "string (UUID, required)",
  "domain": "string (default: 'auto')",
  "client_memory": {
    "data": "string (optional)",
    "signature": "string (optional)"
  }
}
```

- `query`: The raw user question or command.
- `session_id`: The UUID for the current chat session to maintain conversational history.
- `domain`: A hint for the routing engine (e.g., `auto`, `criminal`, `tax`).
- `client_memory`: An HMAC-signed memory object from the frontend `localStorage`. Used for personalizing responses securely.

**Example Request**
```bash
curl -X POST http://localhost:8000/chat \
  -H "Content-Type: application/json" \
  -d '{
        "query": "What is the punishment for theft under BNS?",
        "session_id": "123e4567-e89b-12d3-a456-426614174000",
        "client_memory": {
           "data": "- User is a law student",
           "signature": "abcd1234efgh5678..."
        }
      }'
```

**Response Format (Server-Sent Events)**
The endpoint returns a `text/event-stream` with different event types formatted as JSON strings.

**Event Types:**

1. **`status`**: Indicates the agent's current background action.
   ```text
   data: {"type": "status", "data": "Analyzing request..."}
   data: {"type": "status", "data": "Searching legal database..."}
   ```

2. **`token`**: A chunk of text generated by the LLM. Concatenate these to form the response.
   ```text
   data: {"type": "token", "data": "Under"}
   data: {"type": "token", "data": " section"}
   ```

3. **`memory_update`**: Emitted when the LLM updates the user's personal facts. The client should replace its stored memory with this payload.
   ```text
   data: {"type": "memory_update", "data": {"data": "- User is a law student\n- User is researching theft", "signature": "new_hmac_signature_here"}}
   ```

4. **`metadata`**: Always emitted at the very end of a successful generation. Contains the final clean answer, detected language, and the exact sources used.
   ```text
   data: {"type": "metadata", "data": {
     "answer": "Under section...",
     "label": "BNS",
     "lang": "en",
     "sources": ["Bharatiya Nyaya Sanhita — Section 303: Punishment for theft"],
     "source_type": "documents",
     "route": "agent"
   }}
   ```

5. **`error`**: Emitted if an exception occurs during the stream.
   ```text
   data: {"type": "error", "data": "Pinecone connection timeout"}
   ```

6. **`[DONE]`**: The standard SSE termination string, signifying the connection can be closed.
   ```text
   data: [DONE]
   ```

## HMAC Signature Validation Flow
For the `client_memory` field in `/chat`:
1. **Empty Memory**: If both `data` and `signature` are empty strings, verification passes implicitly.
2. **Verification**: The backend recalculates the SHA-256 HMAC of `client_memory.data` using the server's `MEMORY_SECRET_KEY`.
3. **Comparison**: Using `hmac.compare_digest()`, the server checks if the calculated signature matches `client_memory.signature`.
4. **Failure**: If signatures do NOT match (indicating tampering by the client), the backend logs a warning and proceeds with an empty memory string for that request, completely ignoring the tampered data.
