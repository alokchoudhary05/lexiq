# LexIQ API Reference

This document provides the official specifications for the LexIQ REST API and Server-Sent Events (SSE) interfaces. It outlines the operational endpoints used to interface with the LexIQ AI engine.

## Base URL
**Production**: `[Your Deployed Backend URL]`  
**Local Development**: `http://localhost:8000`

### System Health
#### `GET /health`
Liveness and readiness probe for the LexIQ API. Verifies that the AI engine is initialized and accepting requests.

**Response (200 OK)**
```json
{
  "status": "ok",
  "engine": "LexIQ v3",
  "initialized": true,
  "sessions": 42
}
```

### Streaming Chat Endpoint
#### `POST /chat`
The primary endpoint of the LexIQ platform. Processes user queries through the agentic LLM pipeline and returns a real-time SSE stream containing operational statuses, text tokens, and resolution metadata.

**Request Schema (`ChatRequest`)**
```json
{
  "query": "string (1-2000 chars, required)",
  "session_id": "string (UUID, required)",
  "domain": "string (default: 'auto')",
  "client_memory": {
    "data": "string (optional)",
    "signature": "string (optional)"
  }
}
```

- `query`: The raw input string provided by the user.
- `session_id`: The UUID mapping to the active conversational thread in the database.
- `domain`: Routing hint (`auto`, `criminal`, `tax`).
- `client_memory`: An HMAC-signed memory object utilized for stateless personalization.

**Example Request**
```bash
curl -X POST http://localhost:8000/chat \
  -H "Content-Type: application/json" \
  -d '{
        "query": "What is the punishment for theft under BNS?",
        "session_id": "123e4567-e89b-12d3-a456-426614174000",
        "client_memory": {
           "data": "- User is a law student",
           "signature": "abcd1234efgh5678..."
        }
      }'
```

**Response Format (Server-Sent Events)**
The endpoint responds with `text/event-stream`. Each event is a structured JSON payload.

**Event Types:**

1. **`status`**: Broadcasts the background operations of the AI Agent.
   ```text
   data: {"type": "status", "data": "Analyzing request..."}
   ```

2. **`token`**: A delta text chunk generated by the LLM. 
   ```text
   data: {"type": "token", "data": "Under"}
   ```

3. **`memory_update`**: Dispatched when the LLM modifies the user's personalization facts. The client must overwrite its local cache with this payload.
   ```text
   data: {"type": "memory_update", "data": {"data": "...", "signature": "new_hmac_signature"}}
   ```

4. **`metadata`**: Dispatched exclusively at the termination of a successful generation. Contains the complete synthesized answer, detected locale, and official sources utilized.
   ```text
   data: {"type": "metadata", "data": {
     "answer": "Under section...",
     "label": "BNS",
     "lang": "en",
     "sources": ["Bharatiya Nyaya Sanhita — Section 303: Punishment for theft"],
     "source_type": "documents",
     "route": "agent"
   }}
   ```

5. **`error`**: Dispatched upon system exception or timeout.
   ```text
   data: {"type": "error", "data": "Pinecone connection timeout"}
   ```

6. **`[DONE]`**: Standard SSE termination marker.
   ```text
   data: [DONE]
   ```

## Security & Signature Validation
The `client_memory` field utilizes strict cryptographic validation to prevent payload tampering.

1. **Generation**: Client memory is signed on the server using `HMAC-SHA256` and the platform's `MEMORY_SECRET_KEY`.
2. **Verification**: Upon every `/chat` request, the server recalculates the hash of the incoming `data` string.
3. **Enforcement**: Using `hmac.compare_digest()`, the server asserts the signature. If a mismatch is detected, the memory payload is purged from the execution context to guarantee system integrity.
